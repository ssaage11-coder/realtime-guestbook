# 사용자 프로필 기능 PRD

## 목표

로그인한 사용자가 자신만의 닉네임과 프로필 사진을 설정할 수 있는 환경을 구축한다.

프로필은 방명록 등록자와 댓글 작성자를 식별하는 기본 사용자 정보로 사용한다. 기존 이메일 기반 로그인은 유지하되, 화면에는 이메일 대신 사용자가 설정한 닉네임과 프로필 사진을 우선 노출한다.

## 기능 정의

### 프로필 생성 및 조회

- 회원가입 또는 첫 로그인 후 사용자별 프로필 row를 생성한다.
- 프로필은 `auth.users.id`와 1:1로 연결한다.
- 로그인 사용자는 자신의 프로필을 조회할 수 있다.
- 공개 화면에서 방명록 작성자 또는 댓글 작성자의 닉네임과 프로필 사진을 표시할 수 있도록 최소 공개 조회 범위를 제공한다.
  - 공개 화면 조회 범위는 `user_id`, `nickname`, `avatar_url`로 제한한다.

### 프로필 수정

- 로그인 사용자는 자신의 닉네임을 수정할 수 있다.
- 로그인 사용자는 자신의 프로필 사진을 업로드하거나 교체할 수 있다.
- 프로필 수정은 본인 프로필에만 허용한다.
- 닉네임 또는 프로필 사진 변경 후 기존 방명록/댓글 화면에 반영될 수 있어야 한다.

### 프로필 사진 Storage

- 프로필 사진은 Supabase Storage에 저장한다.
- 파일 경로는 `{user_id}/profile/{timestamp-random}.{ext}` 형식을 사용한다.
- Storage RLS는 첫 번째 path segment가 현재 `auth.uid()`와 일치할 때만 업로드/수정/삭제를 허용한다.
- 프로필 사진은 현재 방명록 이미지와 동일하게 public URL 렌더링을 우선 사용한다.

### UI

- `/profile` 라우트를 추가한다.
- `AuthStatus` 또는 주요 화면에서 프로필 설정 화면으로 이동할 수 있어야 한다.
- 프로필 화면은 현재 손그림 스타일 UI와 일관되게 구성한다.
- 닉네임 입력, 프로필 사진 업로드, 미리보기, 저장 버튼, 로딩/에러/성공 상태를 제공한다.

## 사용자 스토리

- 로그인한 사용자로서, 이메일 대신 내가 정한 닉네임으로 방명록과 댓글에 표시되고 싶다.
- 로그인한 사용자로서, 내 분위기를 보여줄 수 있는 프로필 사진을 설정하고 싶다.
- 로그인한 사용자로서, 언제든 닉네임과 프로필 사진을 바꿀 수 있어야 한다.
- 방문자로서, 방명록 목록과 댓글에서 작성자가 누구인지 닉네임과 사진으로 쉽게 알아보고 싶다.
- 보안 관점에서, 다른 사용자가 내 닉네임이나 프로필 사진을 수정할 수 없어야 한다.

## 필수 요구사항

### 데이터 모델

- `profiles` 테이블을 추가한다.

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| `user_id` | `uuid` | Primary key, `auth.users(id)` 참조 |
| `nickname` | `text` | 사용자 표시 이름 |
| `avatar_url` | `text` | 프로필 사진 public URL |
| `avatar_path` | `text` | Storage 내부 경로 |
| `created_at` | `timestamptz` | 생성시각 |
| `updated_at` | `timestamptz` | 수정시각 |

- `nickname`은 1자 이상 20자 이하로 제한한다.
- `nickname`은 앞뒤 공백을 제거해 저장한다.
- `avatar_url`, `avatar_path`는 프로필 사진 미설정 상태를 허용하기 위해 nullable로 둔다.

### RLS

- `profiles` RLS를 활성화한다.
- 공개 작성자 표시용 조회 표면을 제공한다.
  - 방명록과 댓글에서 닉네임/프로필 사진을 표시하기 위해 `user_id`, `nickname`, `avatar_url`만 노출한다.
  - `avatar_path`, `created_at`, `updated_at`은 공개 작성자 표시 조회에서 제외한다.
- insert/update/delete는 본인만 허용한다.
  - `user_id = auth.uid()`
- 다른 사용자의 프로필 수정은 실패해야 한다.
- 프로필 사진 Storage write 정책은 사용자 ID prefix를 기준으로 제한한다.

### 앱 기능

- `/profile` 페이지를 추가한다.
- 로그인하지 않은 사용자가 `/profile`에 접근하면 `/login?next=/profile`로 유도한다.
- 로그인 사용자는 자신의 현재 프로필 정보를 볼 수 있어야 한다.
- 프로필이 없으면 기본값을 보여주고 저장 시 새로 생성한다.
- 닉네임 저장이 가능해야 한다.
- 프로필 사진 업로드와 미리보기가 가능해야 한다.
- 저장 성공/실패 상태를 사용자에게 표시해야 한다.
- `AuthStatus`에서 이메일만 표시하지 않고, 가능하면 닉네임을 우선 표시한다.

### 기존 화면 연동

- 댓글 작성 폼의 `author_name` 직접 입력 의존도를 줄인다.
- 댓글 작성 시 기본 작성자 이름은 현재 사용자의 프로필 닉네임을 사용한다.
- 프로필 닉네임이 없으면 기존 방식처럼 이름 입력을 요구하거나 이메일 앞부분을 fallback으로 사용한다.
- 방명록 상세 댓글 목록에서 프로필 닉네임과 프로필 사진을 표시할 수 있어야 한다.

### 검증

- 미로그인 사용자는 프로필 저장에 실패해야 한다.
- 사용자 A는 사용자 B의 프로필을 수정할 수 없어야 한다.
- 사용자 A는 사용자 B의 Storage prefix에 프로필 사진을 업로드할 수 없어야 한다.
- 사용자 A가 닉네임/사진을 바꾸면 자신의 프로필 조회 결과에 반영되어야 한다.
- 기존 방명록 등록, 댓글 작성, Realtime 구독이 깨지지 않아야 한다.

## 권장 요구사항

- 회원가입 직후 기본 프로필을 자동 생성한다.
  - DB trigger 또는 앱 첫 로그인 시 upsert 중 하나를 선택한다.
- 닉네임 중복 허용 여부를 명확히 정한다.
  - 초기 버전은 중복 허용을 권장한다.
  - 중복 금지가 필요하면 `lower(nickname)` 기반 unique index를 별도 검토한다.
- 프로필 사진 파일 제한을 추가한다.
  - 허용 MIME type: `image/png`, `image/jpeg`, `image/webp`
  - 권장 최대 크기: 2MB
- 새 프로필 사진 저장 성공 후 이전 프로필 사진 object 삭제를 검토한다.
- 프로필 사진이 없을 때 손그림 스타일의 기본 아바타를 표시한다.
- 댓글 테이블의 `author_name`은 과거 데이터 호환을 위해 유지하되, 신규 댓글은 `profiles.nickname` 기반으로 채우는 방향을 검토한다.
- 목록과 상세 화면에서 작성자 프로필 정보를 효율적으로 가져오기 위해 Supabase join select 또는 별도 profile map 조회 전략을 정한다.
- 프로필 변경을 다른 화면에 즉시 반영해야 한다면 `profiles` Realtime 구독을 추가로 검토한다.
- 운영 전 개인정보 관점에서 공개되는 프로필 필드를 `nickname`, `avatar_url`로 최소화한다.

## 구현 메모

- 2026-05-06: 첫 릴리스 구현을 진행했다.
- `/profile` 라우트에서 로그인 사용자가 자신의 닉네임과 프로필 이미지를 저장할 수 있게 했다.
- 프로필이 없으면 기본 닉네임과 기본 아바타를 표시하고, 저장 시 `profiles` row를 생성한다.
- 방명록 목록과 상세/댓글 화면은 `user_id` 기준으로 `profile_public` 표시 정보를 조회해 닉네임과 아바타를 표시한다.
- 공개 작성자 표시 조회는 `user_id`, `nickname`, `avatar_url`만 사용하며 `avatar_path`, 생성/수정 시각은 공개 화면에 노출하지 않는다.
- 기존 `comments.author_name`은 과거 데이터 fallback으로 유지한다.
- 프로필 변경은 이미 열린 목록/상세 화면에 즉시 전파하지 않고, 이후 로드/재진입 시 반영한다.
- 신규 댓글의 기본 작성자 이름은 현재 사용자 프로필 닉네임을 우선 사용하고, 없으면 이메일 앞부분 또는 기본값을 사용한다.
- 실제 Supabase 프로젝트에는 `supabase/sql/003_profiles.sql` 적용이 필요하다.
